## 修复 App 自动化设备截图 500 错误

**现象**: `POST /api/app-automation/devices/1/screenshot/` 返回 500，前端控制台报 `Failed to load resource: 500` 和 `截图失败`

**根因** (`apps/app_automation/views/device_views.py` screenshot 方法):
1. `subprocess.CalledProcessError` 未单独捕获 — ADB 命令失败时被笼统的 `except Exception` 吞掉，丢失 stderr 错误详情
2. `FileNotFoundError` 未单独捕获 — ADB 路径无效时无法给出明确提示
3. 远程设备（`remote_emulator`）未处理 `device_id` 格式（IP:port）
4. 缺少 Windows `CREATE_NO_WINDOW` 参数

**修复** (后端):
- 新增 `platform` 导入
- 按优先级捕获异常: `FileNotFoundError` → `CalledProcessError`（含 stderr） → `TimeoutExpired` → `Exception`
- 远程设备自动补全 `device_id` 为 `IP:port` 格式
- Windows 添加 `CREATE_NO_WINDOW`
- 日志记录返回码和 stderr

**修复** (前端 `CaptureElementDialog.vue`):
- catch 块展示后端返回的具体错误消息: `error?.response?.data?.msg`

**构建**: ✓ built in 17.04s

```python
elements = []

    elements.extend(self._parse_hierarchy_nodes(child))
```

## 修复: 刷新后text不显示 + 截图底部元素无法点击

**现象**:
1. 后端返回 `"text": "退出登录"` 但前端不显示 text（第一次正常，第二次及以后异常）
2. 描述更正: 不是截图显示不全，而是选择UI层级定位后，截图**下方**的UI元素无法点击选择

### Bug根因

**问题1: text不显示**
- `selectUIElement()` 刷新时通过 bounds 匹配元素 (`Math.abs < 5px`)，会用 `Array.find()` 返回第一个匹配项
- 同一 bounds 可能有多个重叠元素（父容器 + 子 TextView），find() 返回的第一个元素可能是没有 text 的父容器
- 父容器覆盖了 `selectedUIElement`，导致 `v-if="selectedUIElement.text"` 为 false，text 消失

**问题2: 底部无法点击**
- `el-dialog__body` 设置 `overflow: hidden`，阻止了 `.capture-left` 的内部滚动
- `.capture-container` 固定高度与 dialog body 的 `max-height` 不匹配（差40px），导致容器被截断
- 图片底部区域的 hierarchy 覆盖层矩形超出可见范围，无法被点击

### 修复 (`CaptureElementDialog.vue`):

**问题1: 属性丰富度优先匹配**
- 新增 `calcAttrScore(fe)`: 按 resource_id(100) > text(50) > content_desc(30) > clickable(20) > checkable(10) 计算丰富度分数
- bounds 匹配改为收集所有候选元素，然后按分数排序取最高分
- 新增 xpath 精确匹配和 text 内容匹配作为兜底
- 这样即使同一 bounds 有多个元素，优先返回 text 属性最丰富的那个

**问题2: CSS 布局重构**
- `el-dialog__body`: 改为固定 `height: calc(100vh - 145px)` + `display: flex; flex-direction: column`
- `.capture-container`: `height` → `flex: 1; min-height: 0`（自适应填充）
- `.capture-image`: 移除 `max-height` 限制，由 `.capture-left` 的 `overflow: auto` 处理滚动
- 这样图片完整显示，底部元素可通过滚动访问

## 修复: calcAttrScore 评分导致 text 不显示（第三次修复）

**现象**: 前两次修复仍然无效，text 属性始终不显示

**根因**:
`calcAttrScore` 评分 `resource_id`(100) > `text`(50)，导致同一 bounds 下：
- 父容器 LinearLayout: resource_id + clickable = 120 分
- 子 TextView: text="退出登录" = 50 分

父容器得分更高被选中，覆盖了有 text 的子元素，`selectedUIElement.text` 为空 → `v-if="selectedUIElement.text"` 不渲染

**修复** (`CaptureElementDialog.vue`):

### 匹配策略重构
1. **`calcMatchScore(fe, originalElem)` 替代 `calcAttrScore(fe)`**:
   - text: 300 基础分（最高权重）
   - text 与原始元素匹配: +500（关键：确保保留了原始 text 的元素不丢失）
   - resource_id: 100 基础分
   - resource_id 与原始匹配: +200
   - class_name 与原始匹配: +150
   - 原则：text 是最重要的用户可见属性

2. **5 级匹配优先级**:
   - ① 精确 xpath 匹配（最先，最可靠）
   - ② bounds + class_name 精确匹配（防止父容器干扰）
   - ③ bounds 匹配 + 候选排序（带原始元素参考）
   - ④ resource_id 精确匹配
   - ⑤ 全元素 text 精确匹配（兜底）

   - 连接失败时记录详细原因
