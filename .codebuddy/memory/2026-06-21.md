## 修复用例导入功能（3个问题层层排查）

### 问题1: Celery Worker 从未真正启动
- **根因**: `start_celery.bat` 使用了 `--detach` 参数，Windows 不支持（`RuntimeError: This platform does not support detach.`）
- **修复**: 改用 `start /MIN "" venv\Scripts\celery -A backend worker -l info --pool=threads --concurrency=4 --logfile=celery.log`
- 同时改用 `--pool=threads` 避免 Windows 下 prefork 的 PermissionError

### 问题2: 优先级映射不支持 P0/P1/P2/P3
- **现象**: 用户 Excel 中使用 `P0` 等优先级级别，导入服务不支持，172条全部失败
- **修复**: `services.py` PRIORITY_MAP 增加 `p0→critical, p1→high, p2→medium, p3→low`

### 问题3: 不存在的版本引用导致整个行失败
- **现象**: Excel 中引用的版本名如"v2.13迭代"在项目中不存在，导入时 KeyError 崩溃
- **修复**: 版本校验从 `_validate_row` 中移除（不作为硬性错误），导入时只关联存在的版本 (valid_versions 过滤)
- 同时修正 priority/type 校验逻辑：使用 `priority.lower()` 统一比较，移除重复校验

### 最终结果
- 重新派发导入任务：172条全部成功，0失败
- 导入功能现已正常工作

## 批量删除 & 批量修改功能

### 后端新增 API
- **POST `/api/testcases/batch-delete/`** : 批量删除用例，接收 `{"ids": [1,2,3]}`，一次请求完成
  - 只删除用户有权限的项目下的用例
  - 返回 `{deleted_count, not_found_count, message}`
- **POST `/api/testcases/batch-update/`** : 批量修改关联项目和版本，接收 `{"ids": [1,2,3], "project_id": 5, "version_ids": [10,11]}`
  - `project_id` 和 `version_ids` 均为可选，不传则保持原值
  - 自动校验目标和版本是否存在
  - 返回 `{updated_count, not_found_count, message}`

### 新增序列化器 (`serializers.py`)
- `BatchDeleteSerializer`: ids (ListField)
- `BatchUpdateSerializer`: ids + project_id (可选) + version_ids (可选)

### 前端修改 (`TestCaseList.vue`)
- 批量删除：从串行 for 循环改为单次 `POST /testcases/batch-delete/` 请求
- 批量修改：新增"批量修改"按钮和对话框
  - 可选择目标项目和版本
  - 默认值为第一个选中用例的当前参数
  - 切换项目时自动加载对应版本列表（调用 `/versions/projects/{id}/versions/`）
  - 只修改用户明确选择的项，未选择项保持原值

### i18n 新增翻译键
- `batchEdit`, `batchEditTitle`, `batchEditTip`, `batchUpdateSuccess`, `batchUpdateFailed`, `batchDeleteTip`
